# 詳細設計書: AIチャットアプリケーション

## 1. 目的

本書は `requirements.md` の要件に基づき、Firebase 認証済みユーザーだけが利用できる 1 画面 AI チャット Web アプリケーションの詳細設計を定義する。

本設計では、コーディング前に以下を明確化する。

- 画面構成
- 認証フロー
- APIキー管理
- AIプロバイダ接続方式
- モデル自動選択方式
- チャット履歴管理
- エラー処理
- Firebase 公開構成
- 実現可能性とリスク

## 2. 前提

### 2.1 対象ユーザー

- Firebase Authentication に登録済みのユーザー。
- 新規登録画面、ユーザー登録申請画面は提供しない。
- 管理者が Firebase Console 等で事前にユーザーを作成する運用を想定する。

### 2.2 対応プロバイダ

- OpenAI

初期版は OpenAI のみ対応する。Gemini、Claude は将来的な拡張候補とし、初期版の実装対象には含めない。

### 2.3 公開方式

- ソースコードは GitHub で管理する。
- アプリは Firebase Hosting で公開する。
- 認証は Firebase Authentication を使用する。
- Firebase は Spark プランの範囲で利用し、Firebase 側の費用を発生させない。
- Firebase Spark プランで新規デプロイできない Cloud Functions は使用しない。

## 3. 全体アーキテクチャ

### 3.1 推奨構成

```text
Browser
  |
  | Firebase Auth
  v
Firebase Authentication

Browser
  |
  | Provider API request with user API key
  v
OpenAI

Browser
  |
  | read/write chat sessions
  v
Cloud Firestore
```

### 3.2 採用方針

Firebase Spark プランの範囲に収めるため、Firebase Cloud Functions は使用しない。

AI プロバイダ API は、ブラウザに保存したユーザー自身の APIキーを使ってフロントエンドから直接呼び出す。

この方式は Firebase 側の費用を避けられる一方で、APIキー漏洩、CORS、プロバイダ SDK のブラウザ対応制限が大きな論点になる。特に本番公開範囲が広い場合は、Spark だけで安全性と安定性を両立することは難しい。

## 4. 技術構成

### 4.1 フロントエンド

採用:

- Vite + TypeScript
- React
- Firebase Web SDK

理由:

- 1 画面アプリに適している。
- Firebase Authentication の状態監視と画面切替を実装しやすい。
- チャット画面、履歴サイドバー、設定モーダルなどの状態管理がしやすい。
- スマートフォン表示を含むレスポンシブUIを構築しやすい。

### 4.2 バックエンド

Spark プラン前提では、Firebase Cloud Functions などのサーバーサイド API プロキシは使用しない。

フロントエンドから OpenAI の REST API またはブラウザ対応可能な呼び出し方式を直接利用する。

制約:

- Firebase ID token のサーバー検証は行わない。
- AI プロバイダ APIキーはブラウザ内で扱う。
- CORS やプロバイダ側のブラウザ利用可否に影響される。
- サーバー側での APIキー秘匿、リクエスト制御、監査ログ取得はできない。

### 4.3 データストア

候補:

- Cloud Firestore

保存対象:

- チャットセッション
- メッセージ履歴
- 固定・ブックマーク状態
- セッションメタデータ

保存しない対象:

- OpenAI の APIキー

APIキーは要件通りブラウザ保存とする。

## 5. 画面設計

### 5.1 ログイン前画面

目的:

- 未認証ユーザーに対して最小限の案内を表示する。
- 新規登録導線は表示しない。

表示要素:

- アプリ名
- 「ログイン」ボタン
- 認証失敗時のエラーメッセージ

ログイン方式:

- メールアドレス・パスワード認証を使用する。
- Google ログインは初期版では使用しない。
- アプリ内に新規登録フォームは設けない。
- 利用可能なユーザーは Firebase Console の Authentication で管理者が作成したメールユーザーとする。
- ユーザーの追加・削除・無効化は Firebase Console の Authentication で行い、ソースコードや環境変数の変更を不要にする。

### 5.2 メインチャット画面

デスクトップ構成:

```text
+----------------------------------------------------------+
| Header: app name / current user / API status / logout    |
+----------------------+-----------------------------------+
| Session sidebar      | Chat area                         |
| - New chat           | - Message list                    |
| - Session list       | - Waiting indicator              |
| - Pinned sessions    | - Input textarea                 |
| - Bookmarks          | - Send button                    |
+----------------------+-----------------------------------+
```

主要要素:

- セッション一覧
- 新規チャットボタン
- メッセージ表示領域
- 入力欄
- 送信ボタン
- 返答待ちインジケータ
- APIキー設定状態
- モデル選択方針セレクタ
- 自動選択されたモデルの表示

通常利用時の配置方針:

- チャット本文と入力欄を主役にする。
- セッション一覧は 1 行表示を基本としたフラットなリストにする。
- セッション操作アイコンは省スペースにし、通常時は控えめに表示する。
- APIキー設定、接続テスト、モデル方針、モデルカタログなどの補助機能は、折りたたみ式の設定エリアに配置する。
- ユーザー情報、ログアウト、APIキー保存状態などの補助情報は設定エリアに集約し、スマートフォン上部では常時表示しない。
- 直近API応答の単独表示は開発確認用とし、通常利用UIからは外す。
- APIキー読み込み済みなどの通常案内は常時表示せず、送信中・成功・エラーなど行動判断に必要な状態を優先して表示する。

スマートフォン構成:

- 1カラム表示を基本とする。
- 表示順は、チャット、履歴、設定の順を基本とする。
- セッション一覧はチャットの下に配置し、必要に応じてスクロールできるようにする。
- 入力欄と送信ボタンは片手操作しやすい位置・サイズを確保する。
- APIキー設定、モデル選択、ログアウトなどの補助操作はメニュー内に整理する。
- テキスト、ボタン、タグ、メッセージ本文が画面幅からはみ出さないようにする。

API疎通確認用の Phase 0 テスト画面はPCでの確認を主対象とする。ただし、極端な崩れが出ない程度のレスポンシブ対応は行う。

### 5.3 APIキー設定画面

表示方式:

- APIキー未設定時は目立つ設定促進を表示する。
- APIキー設定済みの場合は設定エリア内に控えめに表示する。
- 通常利用時は APIキー設定画面を折りたたみ、チャット履歴とチャット本文の表示領域を優先する。

入力項目:

- OpenAI APIキー

操作:

- 保存
- 削除
- 入れ替え
- 接続テスト
- 静的モデルカタログ表示

表示:

- 未設定
- 設定済み
- 接続確認済み
- 接続失敗
- トークン・クォータ関連エラー
- APIキー無効エラー

## 6. 認証設計

### 6.1 認証状態管理

Firebase Authentication の認証状態を監視し、状態に応じて表示画面を切り替える。

```text
未認証 -> ログイン前画面
認証済み -> メインチャット画面
認証確認中 -> ローディング表示
```

### 6.2 アクセス制御

フロントエンド:

- 未認証時はチャット画面を表示しない。
- AI プロバイダ API 呼び出しは、認証済み画面内の処理として実行する。
- 認証済みユーザーのみアプリ利用可とする。
- サーバー側で Firebase ID token を検証する API は Spark 前提では設けない。

Firestore:

- 認証済みユーザーが自分のセッションだけを読み書きできるルールにする。
- 初期版では、個人利用前提のため Firebase Authentication の認証済みユーザーのみ許可する。
- 親の `users/{uid}` ドキュメント自体はアプリでは使用せず、セッションとメッセージのサブコレクションのみを書き込み対象にする。
- 初期版の Firestore Security Rules では、所有者確認を優先する。
- セッション・メッセージのフィールド名、型、本文サイズの厳密な検証は、初期動作が安定した後の強化項目とする。

## 7. APIキー管理設計

### 7.1 保存方式

保存先:

- ブラウザの IndexedDB

推奨:

- APIキーは IndexedDB に保存する。
- 設定画面上では末尾数文字のみ表示する。
- 削除操作で即時削除する。
- APIキーはチャット、ソースコード、Git 管理対象ファイルに貼り付けない。
- 接続テストでは、実装後のアプリ設定画面にユーザー本人が APIキーを入力して検証する。

補足:

- ブラウザ保存の APIキーは、XSS や端末共有に弱い。
- Web Crypto による暗号化は可能だが、復号鍵も同じブラウザに保存する場合、保護効果は限定的。
- 本番用途で安全性を高めるなら、APIキーをサーバー側で暗号化保存する設計に変更する必要がある。

### 7.2 IndexedDB 保存形式

初期版では OpenAI APIキーのみを保存する。

```text
Database: ai-chat-settings
Object store: secrets
Key: openai-api-key
Value: OpenAI APIキー文字列
```

将来的に複数プロバイダを扱う場合は、プロバイダ別のキー管理形式へ拡張する。

### 7.3 接続テスト

接続テストでは以下を確認する。

- APIキーが形式的に空でない。
- OpenAI API に認証付きリクエストが通る。
- 軽量な疎通確認 API が成功する。
- 失敗時は OpenAI のエラーを共通エラー形式に変換する。

## 8. AIプロバイダ接続設計

### 8.1 共通インターフェース

アプリ内部ではプロバイダ差分を吸収するため、以下の共通インターフェースを定義する。

```ts
type ProviderId = "openai";

type ChatMessage = {
  role: "system" | "user" | "assistant";
  content: string;
};

type ModelInfo = {
  provider: ProviderId;
  id: string;
  label: string;
  policy: "accuracy" | "speed" | "balanced" | "cost";
  featureTags: string[];
  description: string;
};

type ChatRequest = {
  provider: ProviderId;
  model: string;
  messages: ChatMessage[];
};
```

### 8.2 OpenAI

利用方針:

- Responses API を第一候補とする。
- モデル一覧は API から動的取得せず、アプリ内の静的モデルカタログを使用する。
- 初期版ではストリーミングを使用せず、通常レスポンスとして返答を表示する。

### 8.3 静的モデルカタログ

初期版では OpenAI モデルの静的カタログをアプリ内に定義する。

各モデルには以下の特徴情報を持たせる。

- 文章生成向け
- 画像入力に強い
- 推論・深い洞察向け
- 高速
- 低コスト
- バランス型

### 8.4 応答方式

初期版ではストリーミング応答を行わない。

- AI 返答待ち中はインジケータを表示する。
- OpenAI から通常レスポンスが返った後、ユーザー発言と AI 返答を Firestore に保存する。
- 保存されたメッセージは Firestore の購読結果として画面に反映する。
- 過去履歴を開いた場合は、最新の会話が見えるようにメッセージ一覧を末尾へスクロールする。

## 9. モデル自動選択設計

### 9.1 選択基準

ユーザーは以下の方針を選択できる。

- 精度優先
- 速度優先
- コスパ優先
- コスト優先

各方針の意味:

- 精度優先: 時間がかかっても、深い洞察、複雑な推論、詳細な検討を重視する。
- 速度優先: 回答の速さを重視し、短時間で実用的な返答を返せるモデルを優先する。
- コスパ優先: 回答品質、速度、費用のバランスを重視する。
- コスト優先: できるだけ低コストなモデルを優先する。

### 9.2 初期版の選択ロジック

初期版では、質問内容を解析してモデルを自動判別する処理は行わない。

ユーザーが選択した方針に対して、静的モデルカタログから固定マッピングでモデルを選択する。

```text
精度優先: gpt-5.1
速度優先: gpt-5-nano
コスパ優先: gpt-5-mini
コスト優先: gpt-5-nano
```

### 9.3 将来の自動判別ロジック

入力:

- ユーザーが選択した方針
- 設定済み APIキー
- 静的モデルカタログ内の OpenAI モデル
- 質問内容
- 会話履歴の長さ

判定要素:

- 長文・複雑な推論・コード生成: 高性能モデルを優先
- 深い洞察・慎重な比較検討・戦略立案: 精度優先では推論性能の高いモデルを優先
- 短い確認・軽い相談・単純な変換: 速度優先では高速モデルを優先
- 短文質問・要約・軽い相談: 低コストまたは高速モデルを優先
- 会話履歴が長い: コンテキスト長に余裕のあるモデルを優先
- OpenAI APIキー未設定: 送信不可
- 接続テスト失敗中: 警告を表示

### 9.4 将来のモデルスコアリング

```text
score =
  qualityWeight * qualityRank +
  speedWeight * speedRank +
  costWeight * costRank +
  reasoningWeight * reasoningRank +
  contextWeight * contextFit +
  availabilityWeight * availability
```

方針別重み:

```text
精度優先:
  qualityWeight: 高
  reasoningWeight: 高
  speedWeight: 低
  costWeight: 低

速度優先:
  qualityWeight: 中
  reasoningWeight: 低
  speedWeight: 高
  costWeight: 低

コスパ優先:
  qualityWeight: 中
  reasoningWeight: 中
  speedWeight: 中
  costWeight: 中

コスト優先:
  qualityWeight: 低
  reasoningWeight: 低
  speedWeight: 中
  costWeight: 高
```

質問内容による自動判別は、固定マッピングを土台として段階的に拡張する。

### 9.5 モデル定義の管理

モデル名、価格、性能評価は変化しやすいため、初期実装ではアプリ内に静的定義を持たせつつ、後から更新しやすい構造にする。

将来的には以下のいずれかに移行できる。

- Firestore にモデル定義を保存し、管理者が更新する。
- Blaze プランへ移行する場合は、Functions 側でプロバイダ API と価格定義を集約する。
- Remote Config でモデル選択ルールを配信する。

## 10. チャット履歴設計

### 10.1 Firestore コレクション

```text
users/{uid}/sessions/{sessionId}
users/{uid}/sessions/{sessionId}/messages/{messageId}
```

### 10.2 session ドキュメント

```json
{
  "title": "チャットタイトル",
  "createdAt": "serverTimestamp",
  "updatedAt": "serverTimestamp",
  "pinned": false,
  "bookmarked": false,
  "lastMessagePreview": "直近メッセージのプレビュー",
  "provider": "openai",
  "model": "model-id",
  "modelLabel": "model-label"
}
```

### 10.3 message ドキュメント

```json
{
  "role": "user",
  "content": "メッセージ本文",
  "createdAt": "serverTimestamp",
  "provider": "openai",
  "model": "model-id",
  "status": "completed",
  "error": null,
  "sequence": 0
}
```

`sequence` は同一バッチ内で `createdAt` が同時刻になった場合でも、ユーザー発言と AI 返答の表示順を保証するために使用する。ユーザー発言を先、AI 返答を次の連番として保存する。

### 10.4 履歴表示

並び順:

1. 固定セッション
2. ブックマーク済みセッション
3. 更新日時の新しいセッション

操作:

- セッション作成
- セッション選択
- セッション固定・解除
- ブックマーク・解除
- セッション名変更
- セッション削除

セッション削除時は、`users/{uid}/sessions/{sessionId}/messages` 配下のメッセージを削除したうえで session ドキュメントを削除する。初期版では 1 セッションあたり最大 100 メッセージを前提に、クライアント側のバッチ削除で処理する。

固定とブックマークの使い分け:

- 固定: 現在よく使うセッションを一覧上部に常駐させるための操作。
- ブックマーク: 後で見返したい重要セッションに印を付けるための操作。
- 初期版では、固定を最優先、ブックマークをその次、その他を更新日時順に表示する。
- 将来的に検索・絞り込みを追加する場合、ブックマークは重要セッションの抽出条件として利用する。

### 10.5 保存上限

Firebase Spark プランの無料枠と個人利用を前提に、以下を目安の保存上限とする。

- 最大セッション数: 50
- 1セッションあたり最大メッセージ数: 100
- 全体最大メッセージ数: 5,000
- 1メッセージ本文の推奨上限: 20KB

上限を超える場合は、固定・ブックマークされていない古いセッションまたは古いメッセージから整理対象にする。

### 10.6 Firestore Rules

初期版の Security Rules では以下を検証する。

- 認証済みユーザーが自分の `users/{uid}/sessions` と `messages` のみ読み書きできる。
- `users/{uid}` 親ドキュメントへの書き込みは許可しない。
- セッションとメッセージのフィールド検証は、初期動作確認が安定した後に段階的に追加する。

当初は session/message のフィールド名、型、本文サイズも Rules で検証する方針だったが、既存データ互換や `serverTimestamp`、連番フィールド追加時の拒否で動作確認が不安定になったため、初期版では所有者制限を優先する。

## 11. エラー処理設計

### 11.1 共通エラー形式

```json
{
  "provider": "openai",
  "type": "auth_error",
  "message": "APIキーが無効です。",
  "detail": "provider original error summary",
  "retryable": false
}
```

### 11.2 エラー分類

- `auth_error`: APIキー無効、権限不足
- `quota_error`: トークン枯渇、クォータ超過、残高不足
- `rate_limit`: レート制限
- `model_unavailable`: モデル未提供、モデル利用不可
- `network_error`: 通信失敗
- `provider_error`: プロバイダ側エラー
- `unknown_error`: その他

### 11.3 ユーザー通知

API無効:

- 「APIキーが無効、または権限がありません。APIキーを確認してください。」

トークン枯渇・クォータ超過:

- 「利用上限または残高に達している可能性があります。プロバイダの利用状況を確認してください。」

モデル利用不可:

- 「選択されたモデルは利用できません。別のモデルを選択してください。」

通信失敗:

- 「通信に失敗しました。時間をおいて再試行してください。」

## 12. セキュリティ設計

### 12.1 Firebase Authentication

- 未認証ユーザーはアプリ本体を利用できない。
- Firestore Security Rules では `request.auth.uid == uid` のみ許可する。
- Spark 前提では Cloud Functions を使わないため、サーバー側 API での ID token 検証は行わない。

### 12.2 APIキー

要件上、APIキーはブラウザに保存する。

リスク:

- XSS により APIキーを読み取られる可能性がある。
- 共有端末で他ユーザーに利用される可能性がある。
- ブラウザ開発者ツールから確認できる可能性がある。

対策:

- APIキーを DOM に直接埋め込まない。
- 表示時はマスクする。
- Content Security Policy を設定する。
- 外部スクリプトを最小化する。
- APIキー削除機能を明確に提供する。
- プロバイダ側で利用制限、予算上限、キーのローテーションを推奨する。

### 12.3 より安全な代替案

本番セキュリティを優先する場合は、以下の設計に変更する。

- APIキーをブラウザではなくサーバー側に保存する。
- Blaze プランへ移行し、Cloud Functions で暗号化して Secret Manager または Firestore に保存する。
- ユーザーごとのキー参照権限を Firebase Auth UID で制御する。

ただし、この代替案は「APIキーはブラウザに保存」という要件とは異なる。

## 13. Firebase 構成

### 13.1 Firebase Hosting

役割:

- フロントエンド静的ファイルの配信
- HTTPS 公開
- Spark プランの無料枠内での公開

### 13.2 Firebase Authentication

設定:

- メールアドレス・パスワード認証のみ
- 新規登録はアプリ UI から提供しない
- ユーザー追加・削除・無効化は Firebase Console 側で行う
- 初期版で使用しない Google ログインプロバイダは無効化する

### 13.3 Cloud Firestore

用途:

- チャットセッション保存
- メッセージ保存
- 固定・ブックマーク状態保存

### 13.4 Cloud Functions

Spark プラン前提では使用しない。

Blaze プランへ移行する場合の用途:

- AI プロバイダ API プロキシ
- 接続テスト
- 動的モデル一覧取得
- ストリーミング中継
- エラー正規化

### 13.5 Firebase Web 設定

Firebase Webアプリの設定値は `.env.local` に保存し、Vite の `VITE_FIREBASE_*` 環境変数として読み込む。

`.env.local` の実ファイルは Git 管理せず、Firebase Hosting にもアップロードしない。ただし、`VITE_FIREBASE_*` はフロントエンド実行に必要な値であり、ビルド後の JavaScript から参照できる前提で扱う。Firebase Web 設定の `apiKey` は OpenAI APIキーのような秘密鍵ではなく、アクセス制御は Firebase Authentication と Firestore Rules で行う。

利用可能なユーザーは、Firebase Authentication のメールアドレス・パスワード認証で管理する。メールアドレスや許可リストはフロントエンド環境変数に埋め込まない。

ユーザー追加手順:

1. Firebase Console の Authentication でメールユーザーを追加する。
2. 初期パスワードを設定する。
3. 不要になったユーザーは Authentication 側で無効化または削除する。
4. アプリ側には新規登録フォームを設けない。

Git 管理対象には `.env.example` のみ含め、実際の `.env.local` はコミットしない。

## 14. 非機能要件

### 14.1 パフォーマンス

- 初期表示は認証状態確認後、必要なセッション一覧のみ取得する。
- メッセージ本文は選択中セッションのみ取得する。
- セッション一覧では本文全文を取得せず、プレビューのみ使用する。

### 14.2 レスポンシブ対応

- 本番のチャット画面はPCとスマートフォンの両方で利用できることを前提とする。
- デスクトップではサイドバー付きの2カラム構成を基本とする。
- スマートフォンでは1カラム構成を基本とし、セッション一覧や設定は折りたたみ・メニュー化する。
- Phase 0 のAPI疎通確認画面はPCでの利用を主対象とし、スマートフォンでの完全最適化は必須としない。

### 14.3 可用性

- OpenAI APIキーが未設定または無効な場合は、送信を止めて設定画面へ誘導する。
- 選択モデルが利用できない場合は、静的モデルカタログ内の別モデルを選択できるようにする。
- 初期版ではストリーミングを使わないため、通常レスポンスの成功・失敗を明確に表示する。

### 14.4 保守性

- 初期版では OpenAI adapter に集約する。
- 将来プロバイダを追加する場合は provider adapter を分ける。
- モデル選択ロジックは UI から分離する。
- エラー分類は共通化する。

## 15. 実現可能性チェック

### 15.1 全体

条件付きで実現可能。

Firebase Spark プランのみで Firebase 側の費用を発生させない場合、Firebase Authentication、Firebase Hosting、Cloud Firestore の無料枠を使って、認証済みユーザー向けの 1 画面 AI チャットアプリは構築可能である。

ただし、Cloud Functions を使えないため、AI プロバイダ API はブラウザから直接呼び出す必要がある。このため、APIキー保護、CORS、プロバイダ SDK のブラウザ対応に制約がある。

### 15.2 認証

実現可能。

Firebase Authentication は Web アプリで利用でき、ログイン後のみメイン画面を表示する構成にできる。新規登録 UI を出さない運用も可能。

注意点:

- メールアドレス・パスワード認証を使用する。
- 個人利用のため、利用対象ユーザーは Firebase Authentication 側で管理する。

### 15.3 APIキーのブラウザ保存

技術的には実現可能。ただし本番セキュリティ上は注意が必要。

理由:

- ブラウザの IndexedDB に保存できる。
- ただし、OpenAI は APIキーをクライアント側で露出しないことを推奨している。
- OpenAI SDK はサーバーサイド利用を前提にした設計が基本となるため、初期版では REST API 呼び出しを検討する。

推奨判断:

- 個人利用・限定ユーザー向けの内部ツールなら、リスクを明示したうえで採用可能。
- 公開範囲が広い本番サービスなら、サーバー側保存または短命トークン方式に設計変更すべき。

### 15.4 AIプロバイダ直接呼び出し

Spark プラン前提では必要になる。ただし、セキュリティ上は推奨度が下がる。

理由:

- OpenAI は APIキーをブラウザなどのクライアント側コードに露出しないことを推奨している。
- CORS やブラウザ利用可否により、OpenAI 直接呼び出しが不安定になる可能性がある。

推奨判断:

- Firebase に費用をかけないことを優先するなら、直接呼び出しで実装する。
- 安全性と安定性を優先するなら、Blaze プランへ移行して Cloud Functions プロキシを使う。

### 15.5 モデル自動選択

実現可能。ただし完全自動化には運用上の調整が必要。

理由:

- 初期版では OpenAI の静的モデルカタログを使うため、動的モデル一覧取得は不要。
- ただし、価格、性能、コンテキスト長、提供状態は頻繁に変わる。
- 「精度」「速度」「コスパ」「コスト」の判定には、アプリ側のモデルメタデータが必要。

推奨判断:

- 初期実装では静的なモデル定義と簡易スコアリングで開始する。
- 運用しながらモデル定義を更新可能にする。

### 15.6 応答方式

実現可能。

初期版ではストリーミング応答を行わない。ユーザー送信後に返答待ち状態を表示し、OpenAI の通常レスポンスが完了した時点で AI メッセージを表示する。

### 15.7 チャット履歴

Spark プランの無料枠内であれば実現可能。

Cloud Firestore にユーザーごとのセッションとメッセージを保存すれば、過去セッションの呼び出し、固定、ブックマークは実装できる。

ただし、無料枠を超えると Spark プランでは該当プロダクトの利用が停止するため、保存件数、読み取り回数、書き込み回数を抑える設計にする。

### 15.8 Firebase 公開

Spark プランで実現可能。

Firebase Hosting でフロントエンドを公開し、Authentication と Firestore を組み合わせる構成が適している。Cloud Functions は使用しない。

## 16. 主なリスクと対応

| リスク | 影響 | 対応 |
| --- | --- | --- |
| APIキーのブラウザ保存 | キー漏洩、予期せぬ課金 | リスク表示、マスク、削除機能、CSP、プロバイダ側制限 |
| Cloud Functions を使えない | APIキー秘匿やサーバー側中継ができない | Spark では直接呼び出し、必要なら Blaze 移行 |
| Spark 無料枠超過 | Firestore や Hosting が停止する可能性 | 履歴保存量と読み書き回数を抑える |
| モデル価格・名称変更 | 自動選択が不正確になる | モデル定義を更新しやすくする |
| OpenAI API 変更 | 静的モデル定義や呼び出し方式が古くなる | モデルカタログを更新しやすくする |
| OpenAI ブラウザ直接呼び出し | CORS やAPIキー露出の懸念 | 個人利用に限定し、注意表示とキー削除機能を提供 |
| 認証済みユーザー制限 | 意図しないユーザー利用 | アプリに新規登録導線を置かず、Firebase Auth のメールユーザーを管理 |
| クォータ・課金エラー | ユーザー混乱 | エラー分類とわかりやすい通知 |

## 19. 結論

本要件は、Firebase Spark プランの範囲でも条件付きで実現可能である。

ただし、最重要論点は「Cloud Functions を使わず、APIキーをブラウザに保存して直接 AI プロバイダ API を呼び出す」点である。技術的には可能だが、本番セキュリティ上は慎重な扱いが必要となる。

本設計では、Firebase 側に費用をかけないことを優先し、Firebase Hosting、Firebase Authentication、Cloud Firestore を Spark プランの無料枠内で利用する。

セキュリティと安定性をさらに重視する場合は、Blaze プランへ移行し、APIキーをブラウザ保存ではなくサーバー側暗号化保存へ変更することを推奨する。
