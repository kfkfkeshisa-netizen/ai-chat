# 要件定義書: AIチャットアプリケーション

## 1. 概要
- ユーザーが自分の OpenAI APIキーを設定し、AIチャットを利用できる 1 画面の Web アプリとする。
- ソースコードは GitHub で管理し、アプリは Firebase Hosting で公開する。
- Firebase Authentication で認証済みのユーザーのみ利用可能とする。
- アプリ内に新規登録画面やユーザー登録リクエスト機能は設けない。
- Firebase は Spark プランの範囲で利用し、Firebase 側の費用は発生させない。
- Firebase Spark プランで利用できない Cloud Functions は使用しない。
- スマートフォンでの表示・操作も前提としたレスポンシブ UI とする。
- 初期開発中の API 疎通確認機能は PC 表示を主対象とし、スマートフォン最適化は必須としない。

## 2. 認証
- Firebase Authentication を使用する。
- ログイン方式はメールアドレス・パスワード認証とする。
- 未認証ユーザーにはログイン画面のみ表示する。
- ログイン後にメインチャット画面を表示する。
- アプリ内に新規登録画面は設けない。
- 利用可能なユーザーは、Firebase Console の Authentication で管理者が作成したメールユーザーとする。
- ユーザー追加・削除・無効化は Firebase Console の Authentication で行い、ソースコードの変更は不要とする。
- Firebase Console 側では、初期版で使用しない Google ログインプロバイダを無効化する。
- Firebase Console 側で、利用するローカル開発ドメインや Hosting ドメインを承認済みドメインに登録する。

## 3. APIキー管理
- 初期版の対応プロバイダは OpenAI のみとする。
- APIキーはブラウザの IndexedDB に保存する。
- APIキーは Firestore、ソースコード、チャット本文には保存しない。
- OpenAI APIキーは `.env.local` にも保存しない。
- APIキーはアプリ画面からユーザー本人が入力する。
- APIキーは保存、削除、入れ替え可能とする。
- APIキー設定後に OpenAI API の接続テストを実行できる。
- APIキー設定済みの場合は、マスクした状態で控えめに表示する。
- API無効、通信失敗、クォータ・残高・レート制限などのエラーをユーザーに通知する。

## 4. AIモデル選択
- AIモデル一覧は API から動的取得せず、アプリ内の静的モデルカタログとして表示する。
- モデルカタログには、文章生成向け、画像入力、推論・深い洞察、高速、低コスト、コスパなどの特徴を表示する。
- ユーザーはモデル選択方針として以下を選択できる。
  - 精度優先
  - 速度優先
  - コスパ優先
  - コスト優先
- 「精度優先」は、時間がかかっても深い洞察を重視するモデル選択とする。
- 初期版では、質問内容の自動判別は行わず、方針ごとの固定マッピングでモデルを選択する。
- 選択されたモデルはユーザーに表示する。
- 初期版の固定マッピングは以下とする。
  - 精度優先: `gpt-5.1`
  - 速度優先: `gpt-5-nano`
  - コスパ優先: `gpt-5-mini`
  - コスト優先: `gpt-5-nano`
- 将来的な拡張候補として Gemini、Claude を想定するが、初期版には含めない。

## 5. チャット機能
- AIと通常応答形式で会話可能とする。
- 初期版ではストリーミング応答は行わない。
- 返答待ち状態を視覚的に表示する。
- チャット入力欄は通常利用を前提とし、疎通確認文を初期入力しない。
- 過去履歴を開いたときは、最新の会話が見えるようにメッセージ一覧を末尾へスクロールする。

## 6. チャット履歴
- チャット履歴は Cloud Firestore に保存する。
- 過去のチャットセッションを選択・呼び出し可能とする。
- チャットセッションを新規作成可能とする。
- よく使うセッションは固定可能とする。
- 後で見返したいセッションはブックマーク可能とする。
- チャットセッション名を変更可能とする。
- 不要なチャットセッションを削除可能とする。
- セッション削除時は、そのセッション配下のメッセージも削除する。
- メッセージ表示順は、ユーザー発言、AI返答の順序が崩れないように制御する。
- 保存上限は、最大50セッション、1セッションあたり最大100メッセージ、全体で最大5,000メッセージを目安とする。
- 上限を超える場合は、固定・ブックマークされていない古い履歴から整理する。

## 7. Firebase / Firestore 制約
- Firestore Security Rules では、認証済みユーザーが自分の `users/{uid}` 配下のみ読み書きできるようにする。
- 初期版では個人利用前提のため、Firebase Authentication で認証済みのユーザーのみ Firestore を利用できるようにする。
- `users/{uid}` 親ドキュメントへの直接書き込みは許可しない。
- 初期版では、セッション・メッセージの詳細なフィールド検証よりも、所有者制限と安定動作を優先する。
- 詳細なフィールド検証は、初期版の動作が安定した後の強化項目とする。

## 8. レスポンシブ UI
- 本番のチャット画面は PC とスマートフォンの両方で利用できることを前提とする。
- デスクトップでは、履歴サイドバーとチャット領域の 2 カラム構成を基本とする。
- スマートフォンでは 1 カラム構成を基本とし、履歴・設定・補助情報が操作の邪魔にならないよう整理する。
- スマートフォンでは、表示順をチャット、履歴、設定の順にする。
- APIキー設定、接続テスト、モデル方針、モデルカタログは通常利用時に目立ちすぎないよう、折りたたみ式の設定エリアへ配置する。
- ユーザー情報、ログアウト、APIキー保存状態などの補助情報は設定エリアへ寄せ、スマートフォン上部には常時表示しない。
- 直近API応答の単独表示は開発確認用とし、通常利用UIには表示しない。
- 通常の案内メッセージは画面内で目立たせすぎず、成功・エラー・送信中などユーザーの判断に必要な状態を優先して表示する。
- チャット履歴は一覧性を優先し、1行表示を基本としたフラットなリストにする。
- 固定、ブックマーク、名前変更、削除などの履歴操作は省スペースなアイコン操作とする。

## 9. 現在の実装済み範囲
- React + Vite + TypeScript のアプリ基盤。
- Firebase Web 設定の `.env.local` 読み込み。
- メールアドレス・パスワードログイン。
- OpenAI APIキーの IndexedDB 保存、削除、読み込み。
- OpenAI Responses API へのブラウザ直接呼び出し。
- 接続テスト。
- 静的モデルカタログ。
- 方針別の固定モデル選択。
- Firestore へのチャット履歴保存。
- 履歴の再読み込み・復元。
- セッション固定、ブックマーク、名前変更、削除。

## 10. 今後の候補
- 通常利用しやすい UI 整理。
- スマートフォン表示の詳細調整。
- セッション検索・ブックマーク絞り込み。
- 履歴保存上限に達した場合の自動整理。
- Firestore Rules のフィールド検証強化。
- CSP などのセキュリティヘッダー強化。
- 必要に応じた Blaze プラン移行と Cloud Functions プロキシ化。
